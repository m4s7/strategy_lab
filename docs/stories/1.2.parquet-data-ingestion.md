## Dev Notes
This story implements the data ingestion foundation for processing MNQ futures tick data from Parquet files. Critical for handling massive datasets efficiently.

**Data Architecture Context:**
- **Data Location**: ./data/MNQ/ with contract month subdirectories
- **File Organization**: Contract months like "03-20", "06-24", "03-25" containing daily Parquet files
- **Data Scale**: 15M+ ticks per day, millions of Level 2 operations, nanosecond timestamps
- **Memory Target**: Process within 64GB RAM constraint

**MNQ Data Schema (from CLAUDE.md):**
- **level**: Data level indicator (string)
- **mdt**: Market Data Type (0=Ask, 1=Bid, 2=Last, 3=DailyHigh, 4=DailyLow, 5=DailyVolume, 6=LastClose, 7=Opening, 8=OpenInterest, 9=Settlement)
- **timestamp**: Nanosecond precision timestamps
- **price**: Decimal128 with 13 digits, 2 decimal places
- **volume**: Int32 trade/quote volume
- **operation**: Order book operations (0=Add, 1=Update, 2=Remove)
- **depth**: Order book depth level
- **market_maker**: Market maker identification

**Technical Implementation Requirements:**
- **Source Module**: src/strategy_lab/data/ingestion/
- **Key Components**: parquet_loader.py, data_validator.py, file_discovery.py
- **Performance**: Target 1M+ ticks/second loading speed
- **Memory Optimization**: Use pandas chunking, pyarrow for efficient I/O

**Data Quality Standards:**
- Timestamp sequences must be monotonic increasing
- Price values within reasonable MNQ trading ranges
- Volume values must be positive integers
- Missing data gaps clearly identified and reported

**Error Handling Requirements:**
- Graceful handling of corrupted files
- Clear error messages for schema mismatches
- Retry logic for I/O failures
- Comprehensive logging for debugging
